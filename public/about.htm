Salsa OAI - Minimalistic OAI data provider with Dancer

About Salsa OAI

Salsa OAI is a simple OAI 2.0 data provider.

Currently supported

  • All six OAI v2.0 verbs should work in a rudimentary way.
  • All error as described in the specification work. I still need to check that.

Not (yet) supported

  • Specification
    • TODO: Check that all error messages demanded by the specification are supported.
    • resumptionToken
    • hierarchical Sets
  • Server setup / Salsa_OAI architecture
    • Stream response. Should speed up listRecords and listIdentifiers A LOT ;-)

Components

Salsa OAI has several components: a backend, a frontend and a data store engine.

Engines

First version (which could easily be resurrected) avoided a database. This version uses sqlite as a database. It should soon be easy to plugin different database engines of your choice. Salsa OAI is written in perl, available on GitHub and uses T. Brody's excellent HTTP::OAI as well as Dancer, a new and cool web framework with a vibrant comunity (See http://perldancer.org ). Salsa OAI is written by Maurice Mengel for the Ethnological Museum in Berlin and MIMO (see http://mimo-project.eu).

Metadata formats

Salsa OAI is mostly agnostic about the metadata formats used. You can use whatever you like as the native xml format as long as you provide a XSLT 1.0 transformation to the desired output format. How cool is that?

SQLite

New version works with SQLite.

Avoiding a database: memory and disk instead

Using its simple engine, Salsa OAI reads OAI header information directly into memory. It uses a file cache to do so, which speeds up extracting the OAI header information from the records. This solution is NOT sustainable (i.e. will not scale well when your repository grows). But with a few thousand records it might work surprisingly well. The motto is: just one notch up from a static repository. Of course, the number of records you can serve comfortably depends mainly on the memory of your server.

I experimented with a file cache for ListRecords to save the potentially big result document on disk. It sort of worked.

Performance

Salsa OAI is written for easy maintenance and extensibility, not for speed. It is very possible that it is too slow for you. Should be in the implementation specific stuff

I provide some performance statistics, so you can see for yourself.
